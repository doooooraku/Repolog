### (B) 最大のボトルネック「PDF関連バグ修正」のなぜなぜ分析

```
問題: PDF関連のバグ修正に約10日費やした（全開発期間の14%）
│
├─ なぜ1: なぜPDF修正にこれほど時間がかかったのか？
│  → OOM、blank PDF、hang、空白ページ挿入が「異なる原因の異なる症状」として次々に出現した
│  │
│  └─ なぜ2: なぜ一度に全部出なかったのか？
│     → 上層のバグ（フォント埋め込み 15MB）を修正すると、隠れていた下層のバグ（CSS grid×大画像 hang）が露出する
│     → レイヤードバグ構造だった
│     │
│     └─ なぜ3: なぜレイヤードバグの存在を事前に予測できなかったか？
│        → expo-print（OS ネイティブ橋渡し API）の境界条件が未知だった
│        → 「HTML サイズ × 画像枚数 × CSS構造」の組み合わせ閾値はドキュメントに記載がない
│        │
│        └─ なぜ4: なぜ事前に boundary testing をしなかったか？
│           → 実機テストが 2/27（開発開始から28日後）まで行われなかった
│           → CI はJest（ロジックテスト）のみで、ネイティブ API の実動作テストがなかった
│           │
│           └─ なぜ5: なぜ実機テストが遅れたか？
```

## 次のアプリへの申し送り TOP5

### 1. 実機テストは最速で行う（Phase 3.5）

**Repolog で最もコストが高かったミス**。コア実装後 28 日間、実機テストをしなかった結果、PDF OOM / blank PDF / hang / 空白ページ挿入が次々に発覚し、修正に 10 日以上を費やした。native bridge API（expo-print, expo-image-manipulator 等）の境界条件は、エミュレータや Jest では絶対に発見できない。**3日以内に実機にインストールする**ことを Phase 3.5 として設定する。

### 2. native bridge API は「壊れる前提」で設計する

`Print.printToFileAsync` は resolve も reject もせず hang する。`expo-image-manipulator` はストレージ不足で書き込みが hang する。native bridge API は「正常に動く」前提ではなく、「タイムアウト + フォールバック + 構造化ログ」の 3 点セットで設計する。具体的には:
- `Promise.race([api, timeout])` でタイムアウト
- フォールバックチェーン（full → reduced → tiny）
- 構造化ログ（入力サイズ、処理時間、結果バイト数）

### 3. ストア申請の準備は開発初日に始める

Google Play サービスアカウントの権限反映に 48 時間、ASC API Key の取得、`usesNonExemptEncryption` の設定、12 人テスター要件の確認 ── これらはすべて開発初日に着手できる。最終日に「あれがない」「これが通らない」で数日ロスしない。

### 4. ADR の前提条件は CI テストに落とす

ADR-0009（PDF slack 3.265mm）は「photo-no が in-flow ゼロ」という前提で成立していたが、PR #212 でその前提が壊れ、iOS 空白ページが再発した。**ADR の前提条件を人間のレビューに頼らず、CI テストで機械的に守る**。

### 5. 観測性（構造化ログ）は初日から投資する

PR #293 で導入した構造化ログ（cssBytes / htmlBytes / sizeBytes / strategy）が、その後の全 PDF バグ調査で決定打になった。native bridge API を使う機能には、最初から構造化ログを設計する。投資対効果は複利で効く。

> **テスト戦略の穴**: Jest 256テストは構造テスト・ロジックテストとして優秀だが、**実機でのランタイム挙動テストがゼロ**。PDF OOM、hang、blank PDF はすべて実機でしか発見できなかった。
>
> **手動 QA の形骸化**: ADR-0009 で定義した手動 QA チェック（次回リリース前に確認）は実際には実行されず、SSoT ドリフトを見逃した。QA 項目は「機械で実行可能な形」に書くか、CI に組み込む。
>
> **提案**: Maestro で PDF 出力フロー（0枚/10枚/50枚）の E2E テストを追加する。出力されたPDFの `/Count` をスクリプトで検証すれば、空白ページ挿入を自動検出できる。

### 4. DevOps エンジニア

> **CI/CD は成熟度が高い**: 5ゲート verify、SHA pin ポリシー、i18n チェック、config チェック、fastlane 自動化。個人開発としては過剰なくらい良い。
>
> **EAS 環境変数のトラップ**: ローカルビルドは `.env` を読むが EAS クラウドビルドは読まない。`prebuild-env-check.mjs` で防御したのは良い対応。テンプレートに含める。
>
> **fastlane の 7つの教訓**: L-FL01〜FL07 は再利用性が高い。特に「precheck は `default_rule_level: :warn` のみ」「`sync_screenshots` は使わない」は次のプロジェクトで即座に活きる。

### 5. プロダクトオーナー

> **価値定義が明確**: 「現場の写真を提出に強い PDF にまとめる」という 1 行のコア価値がブレなかった。Non-goals（クラウド同期、承認フロー等）も明確。
>
> **Free/Pro の境界設計が良い**: プレビュー無制限 + 出力制限という設計は、ユーザーに価値を体験させてから課金に導く理想的な構造。
>
> **懸念**: PDF 出力の品質問題（attempt 1 失敗→reduced fallback で画質低下）が Pro ユーザーの解約理由にならないか。ADR-0016 の再評価条件を定期的にチェックすべき。

### 6. 個人開発者の先輩

> **70日で両プラットフォーム・19言語は素晴らしい**。AI 駆動開発の威力を最大限に引き出している。
>
> **ただし、PDF に時間を使いすぎた**。全開発期間の 14% を PDF バグ修正に費やしている。次のアプリで PDF 生成が必要なら、expo-print ではなく react-native-pdf-lib や pdf-lib（JavaScript で直接 PDF を構築）を検討する価値がある。OS のレンダリングエンジンに依存しない分、予測可能性が高い。
>
> **テンプレートの価値**: app_template リポジトリに Repolog の全教訓（CI, i18n, ストア申請チェックリスト, prebuild check, postbuild verify）を組み込めば、次のアプリは **20日で両ストアリリース** が見えてくる。それが「アプリファクトリー」の本質。
>
> **一つだけ**: ドキュメントを書きすぎている印象がある。lessons.md は 300 行超、ADR が 18 件。個人開発では「テンプレート + ADR 5件以内 + lessons は箇条書き」で十分。次のアプリでは docs のボリュームを 1/3 に絞ることを推奨する。
